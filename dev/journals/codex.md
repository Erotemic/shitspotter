* When a composed model fails but one stage is clearly healthy, the next experiment should freeze the healthy stage and turn the failing stage into a controlled comparison.
* Debug loops get dramatically better once intermediate prompt metadata is carried through into artifacts instead of being discarded as transient implementation detail.
* If a postprocess parameter can plausibly erase the signal you are trying to measure, it needs to be first-class in the CLI surface, not buried in package defaults.

The first live `v2` run provided useful evidence and a very ordinary failure mode at the same time. Before the script crashed, it already answered two high-value questions: the validation and test datasets have exact image-size agreement with what kwcoco thinks they are, so there is no obvious width/height swap or EXIF-style orientation mismatch at the dataset level; and the GT-box prompted SAM2 `tuned_raw` condition still evaluated to essentially zero AP, which is strong evidence that the detector is not the main bottleneck and that raw postprocess alone is not rescuing the tuned segmenter. The crash was on me rather than on the model logic: the geometry-report helper inside the shell script treated the report path as a string and then called `.parent` on it, which only surfaced once the first GT-box run successfully completed and tried to emit its report. I fixed that immediate bug by converting the report path to a `Path`, but I also stepped back and made the script resumable. The original script inherited the “never overwrite” mindset from the experiment scripts, which is usually a good safety default, but in a long diagnostic matrix that behavior becomes brittle: one helper bug can force an entirely new experiment label even though most of the expensive predictions already succeeded. The revised `v2` script now reuses existing packages, dataset reports, and eval outputs when the expected files are already present, so rerunning after an interruption should continue rather than demand cleanup or a new version number.

I feel good about that tradeoff. For canonical train/eval generation scripts, strict overwrite guards still make sense because version labels are part of the provenance story. For debug-oriented matrix scripts, resumability is more valuable than pristine first-run semantics because the real cost is GPU time and operator attention, not conceptual purity. The remaining uncertainty is scientific rather than infrastructural: once the resumed run finishes, we still need to interpret whether zero-shot SAM2 meaningfully outperforms tuned SAM2 and whether the prompt-vs-mask geometry summaries point toward a prompting bug or just a poor segmenter.

Design takeaways:
* Diagnostic scripts should usually be resumable, even when top-level experiment scripts are intentionally immutable.
* A crash in report generation can still leave the underlying predictions valuable, so skip/reuse behavior is worth adding early for long matrix runs.
* Narrowing the hypothesis space is progress even when the measured metric stays at zero; “it still fails under GT boxes with raw postprocess” is a meaningful experimental result.
